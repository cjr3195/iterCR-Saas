## 🛠️ System Architecture

IterCR was engineered as a B2B2C SaaS platform to resolve the friction between school transport companies and families. To ensure high availability and mobile accessibility without forcing users to download native apps, it was structured as a **Progressive Web App (PWA)**.

**Core Tech Stack:**
* **Frontend & Serverless Backend:** Next.js (React), Tailwind CSS
* **Database & Auth:** Supabase (PostgreSQL)
* **Real-time Engine:** WebSockets
* **Push Notifications:** OneSignal

---

## 🚀 Key Engineering Highlights & Data Flows

### 1. Trip State Machine & Conditional Telemetry
To prevent unnecessary bandwidth consumption and reduce database reads/writes, the telemetry system operates on a strict state machine. GPS coordinate broadcasting is only active when a trip transitions to the `"En curso"` (In Progress) state and completely halts upon reaching `"Finalizado"` (Finished).

### 2. Geofencing & Automated Push Notifications
Integrated with the **OneSignal API**, the system calculates the distance between the transport's live coordinates and the specific geofences established for each student's drop-off/pick-up point. When the transport breaches a geofence radius, mathematical evaluations trigger automated Push Notifications to the parents' devices, delivering accurate Estimated Times of Arrival (ETA).

### 3. Absolute Privacy & "Single-Use Tokens"
Because the platform handles sensitive data (minors' locations), data privacy was the top priority. Parents access the live tracking dashboard via encrypted, single-use tokens generated by the transport company. There are no manual passwords to manage, and the tracking link automatically expires the moment the trip concludes, closing any security vulnerabilities.

## 📊 Relational Database Design

A NoSQL approach was discarded in favor of **PostgreSQL** to maintain strict referential integrity between Transport Companies, Drivers, Routes, and Students. 

To handle the high frequency of GPS pings without degrading the performance of the core application, the database architecture heavily separates concerns:
* **Static Entities:** Standard relational tables for users, roles, and route configurations.
* **Telemetry Storage:** High-write optimized tables tailored for sequential time-series data, enabling accurate historical auditing of routes and speeds.
